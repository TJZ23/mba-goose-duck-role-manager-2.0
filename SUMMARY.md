## 🔄 核心改进

### PeerJS 方案的局限性
```
问题：
├─ 需要指定一人作为"房主"
├─ 房主必须始终在线
├─ NAT 穿透复杂，容易失败
├─ 延迟取决于两人的网络质量
└─ 无法扩展到多人（点对点的限制）
```

### Socket.io 方案的优势
```
优势：
├─ ✅ 中央服务器管理房间（房主无需特殊角色）
├─ ✅ 房主可离线，其他人继续同步
├─ ✅ 服务器自动处理 NAT 穿透
├─ ✅ 延迟可控，由服务器配置决定
├─ ✅ 轻松支持多人（无点对点限制）
└─ ✅ 生产级别可靠性
```

## 🎯 网络同步工作原理

### 实时同步流程

```
用户 A                  服务器                    用户 B
  │                       │                         │
  ├─── 点击按钮 ───────────────►                  │
  │   (本地立即反馈)                               │
  │                       │                         │
  │                   接收操作                      │
  │                   ├─ 更新状态                  │
  │                   ├─ 转发给房间               │
  │                       │─────── 广播操作 ──────►│
  │                       │                   同步 UI
  │                       │                         │
  │◄─── 广播回传 ───────────                      │
  │   (验证结果)                                   │
  │                       │                         │
  └─── 一起显示同步状态 ─────────────────────────┘
```

**关键特点：**
- 用户点击后**立即反馈**（乐观更新）
- 不等待服务器确认
- 最终状态 **0-500ms** 时间内所有人一致
- 完全避免了竞态条件

## 🔧 关键函数变化

### 旧版（PeerJS）
```javascript
// 初始化 Host
globalPeer = new Peer(getRoomKey(myRoomId));
globalPeer.on('connection', (conn) => { ... });

// 初始化 Client
hostConnection = globalPeer.connect(getRoomKey(targetRoomId));
hostConnection.send({ type: 'STATE_UPDATE', payload: state });
```

### 新版（Socket.io）
```javascript
// 初始化 Socket
socket = io();
socket.on('connect', () => {
  socket.emit('join-room', { roomId, userId });
});

// 发送操作
socket.emit('user-action', { roomId, action, payload });

// 接收同步
socket.on('state-updated', (data) => {
  applyRemoteState(data.state);
});
```

## 🆘 常见问题解决

| 问题 | 解决方案 |
|------|---------|
| 加入房间失败 | 清除 localStorage，刷新页面，检查服务器是否运行 |
| 数据未同步 | 检查网络连接，查看浏览器控制台日志，重启服务器 |
| 端口被占用 | 改用其他端口：`PORT=8080 npm start` |
| 跨域错误 | 确保访问地址与服务器一致 |

详见 [TEST.md](./TEST.md) 的故障排除部分。
